## 7. 第 3 页：ChatGPT 与 Codex 的区别

### 核心观点

ChatGPT 更像通用顾问，适合讨论、解释、写作和规划。Codex 更像工程协作者，适合进入代码项目，修改文件、运行命令、修复问题。两者是分工关系，不是替代关系。

## 8. 第 4 页：VPN 与网络环境

### 标题

稳定、合规的网络环境

### 核心观点

AI Agent 工作依赖稳定连接、账号安全和合规使用。网络环境不是越复杂越好，而是要稳定、安全、可持续。

### 展示内容

* 稳定性：减少中断，避免长任务执行到一半断线。
* 合规性：遵守学校、实验室、平台和当地规则。
* 安全性：不要使用来源不明的客户端，不要泄露账号、Cookie 或 token。
* 隐私性：不要让敏感论文数据、未公开代码或账号信息经过不可信服务。
* 可维护性：优先选择自己能长期理解和管理的方案。

### 案例

运行 Codex 处理论文图表脚本时，如果网络频繁中断，可能导致依赖安装、文档检索或远程工具调用失败。应先确认网络稳定，再启动长任务。

### 展示方式

使用“网络准备清单”卡片。不推荐具体商业服务，除非后续提供公开可展示链接。

## 10. 第 6 页：Codex 使用技巧

### 标题

让 Codex 稳定工作的关键

### 核心观点

Codex 不只是“回答问题”，而是可以进入项目、执行任务、运行命令和验证结果。给 Codex 的任务越清楚，结果越稳定。

### 展示内容与案例

1. 目标：说明想让 Codex 完成什么。  
   案例：请把这段数据处理脚本重构成函数，并保持输出一致。

2. 上下文：提供文件、报错、论文背景或数据说明。  
   案例：把 MATLAB 报错、输入数据格式和目标图形一起给 Codex。

3. 约束：限定不要做什么。  
   案例：这次只改绘图样式，不改数据计算逻辑，不碰部署配置。

4. 验收：说明任务完成后如何确认。  
   案例：修改网页后运行 `npm run build`，并总结改了哪些文件。

5. 先计划再执行：复杂任务先让 Codex 访谈和规划。  
   案例：制作毕业论文图表自动化流程前，先让 Codex 拆分步骤和风险。

### 展示方式

使用技巧卡片，每张卡片包含“技巧”和“案例”。

## 12. 第 8 页：Skills 的安装、制作与检索

### 标题

把重复流程沉淀为 Skills

### 核心观点

Skills 是可复用工作流。它们把某类任务的步骤、注意事项和资源打包起来，让 Codex 在遇到相似任务时更稳定。

### 展示内容与案例

1. 安装：用 `$skill-installer` 安装可复用技能。  
   案例：安装论文阅读、PPT 制作或表格处理相关技能。

2. 制作：用 `$skill-creator` 把重复流程沉淀成技能。  
   案例：把“论文图表统一美化导出”的流程做成自己的 skill。

3. 检索：用 `$技能名` 或任务描述让 Codex 自动匹配。  
   案例：输入“用论文图表 skill 优化这张图”，让 Codex 读取对应 SKILL.md。

### 展示方式

使用三张卡片，每张卡片包含动作、说明和案例。

## 13. 第 9 页：MCP 与工科研究生工作流

### 标题

让 Codex 连接 MATLAB 与研究工具

### 核心观点

MCP 让 Codex 连接外部工具和上下文。对工科研究生来说，MCP 的价值在于把代码修改、计算验证、图形输出和上下文说明连起来。

### 展示内容

* MCP：连接模型与工具、文档、数据或软件环境。
* MATLAB MCP：让 Codex 调用 MATLAB，运行脚本、检查报错、读取输出。
* 研究场景：数值模拟、实验数据处理、论文图表生成、批量参数扫描。
* 优势：减少手动复制报错，保留上下文，让修改和验证形成闭环。

### 案例

用户提供 MATLAB 脚本和报错后，Codex 通过 MCP 调用 MATLAB 运行脚本，定位变量维度问题，修改代码，再重新运行验证图形是否正常输出。

### 展示方式

使用“研究工作流”卡片或流程图。

## 14. 第 10 页：CLI 是什么、有什么用、怎么用

### 标题

CLI：终端里的 Codex

### 核心观点

CLI 是命令行界面。Codex CLI 让 Codex 在终端中进入本地项目，读取文件、修改代码、运行命令并恢复会话。

### 展示内容

* 进入项目：在项目目录运行 `codex`。
* 带任务启动：运行 `codex "Explain this codebase to me"`。
* 恢复会话：运行 `codex resume --last`。
* 适用场景：本地代码解释、批量修改、构建验证、脚本调试。

### 案例

在一个数据处理项目中，先运行 `codex "Explain this codebase to me"` 让 Codex 解释目录结构，再让它修复报错并运行测试。

### 展示方式

使用命令卡片，不写长段代码。

### 核心观点

GitHub 负责代码留痕、分支协作和回滚。Netlify 负责自动构建、预览和上线。

### 案例

Codex 修改网页内容后，先本地构建，再提交到 GitHub。Netlify 自动生成 Deploy Preview，用于检查正式页面效果。

### 核心观点

## 17. 第 13 页：人类如何引导 Agent

### 标题

人类不是旁观者，而是导演

### 核心观点

Agent 可以执行任务，但方向、边界和判断仍需要人类引导。

### 示例指令

* 先不要改代码，先给我计划
* 这次只修改视觉样式，不要改内容
* 请解释为什么要改这个文件
* 请把任务拆成三个小步骤
* 请运行构建并根据报错修复

### 案例

如果 Codex 改动范围过大，人类可以要求它暂停、说明差异、缩小范围，并只保留与目标直接相关的修改。

### 展示方式

使用对话气泡或提示词卡片。

## 18. 第 14 页：Fork / Worktree

### 标题

让多个任务安全并行

### 核心观点

Fork 和 Worktree 都能降低修改风险，但它们服务于不同场景。这里重点讲 Codex app 中的 Worktree 工作方式。

### 展示内容

* Fork：复制远程项目，用于独立探索或参与开源项目。
* Worktree：Codex app 中用于并行任务的独立工作现场。
* Handoff：把线程和代码在 Local 与 Worktree 之间安全移动。
* 适用场景：一个任务改页面内容，另一个任务检查构建问题，互不干扰。

### 案例

同时开两个 Codex 线程：一个在线程 A 中优化网页文案，另一个在线程 B 中检查样式和构建。确认无冲突后，再把需要的线程交接回 Local。

### 展示方式

使用两列对比卡片，避免把 Branch 作为独立主题展开。

## 19. 第 15 页：总结

### 标题

AI Agent 工作流

### 三个重点

1. 说清楚：目标、上下文、约束、验收。

真正重要的不是让 AI 替我们完成所有事，而是学会把复杂任务拆成可执行、可验证、可回滚、可复用的工作流。
